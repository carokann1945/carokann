# Carokann

<br />

## 📌 개요

반복 작업을 완료해도 즉시 체크가 사라지지 않아 작업의 진척도를 직접 확인할 수 있도록 만든 개인용 작업 리셋 트래커입니다. 탭 단위 분류, 다양한 반복 주기, 로컬 모드와 로그인 저장을 함께 제공해 루틴과 반복 작업을 관리할 수 있습니다.

**참여 인원** : 1인

**기간** : 2026.03 ~ 현재

**Live** : https://carokann.app/

**GitHub** : https://github.com/carokann1945/carokann

<br />

## 🤔 개발 동기

기존 작업 관리 서비스들은 반복 작업을 완료하면 체크가 즉시 해제되거나 다음 상태로 전환되어 이번 주기 안에서 내가 얼마나 진행했는지를 직관적으로 확인하기 어렵다는 아쉬움이 있었습니다. 특히 게임에서의 루틴처럼 한 번의 완료 여부보다 "이번 주에 3번 했는지", "이번 달 목표를 얼마나 채웠는지"가 더 중요한 작업에서는 이런 방식이 적합하지 않다고 느꼈습니다.

Carokann은 이 문제를 해결하기 위해 만들었습니다. 사용자가 매일, 매주, 매월, 매년뿐 아니라 원하는 간격의 커스텀 주기까지 자유롭게 설정할 수 있도록 했고, 반복 작업의 체크 기록이 다음 리셋 시점까지 유지되도록 구성해 현재 사이클의 진행도를 눈으로 확인할 수 있게 했습니다. 단순히 작업을 지우고 다시 만드는 방식이 아니라 반복되는 작업을 더 명확하게 추적하는 경험 자체를 목표로 설계한 서비스입니다.

<br />

## ✨ 주요 기능

**반복 작업 관리**

- 일반 작업과 반복 작업을 분리해 용도에 맞게 관리할 수 있습니다.
- 반복 작업은 `매일`, `매주`, `매월`, `매년`, `커스텀 N일` 주기를 지원합니다.
- 한 사이클 안에서 달성해야 할 체크 횟수를 설정해 다중 체크 목표를 관리할 수 있습니다.
- 다음 리셋 시점과 남은 시간을 표시해 언제 다시 시작되는지 바로 확인할 수 있습니다.
- 시간대 기준 계산을 지원해 `Plain` 또는 브라우저 시간대 기준으로 리셋 시점을 제어할 수 있습니다.

<br />

**저장 및 계정 흐름**

- 로그인 없이도 바로 사용할 수 있는 비회원 모드를 제공합니다.
- 비회원 상태에서는 브라우저 로컬 스토리지에 데이터를 저장합니다.
- Google 계정으로 로그인할 수 있으며, 인증은 Supabase Auth 기반으로 처리합니다.
- 최초 로그인 시 로컬 데이터를 Supabase DB로 이관해 기존 작업 흐름을 이어갈 수 있습니다.
- 로그인 이후에는 서버에 데이터를 저장해 다른 기기에서도 같은 상태를 이어서 사용할 수 있습니다.

<br />

**사용성**

- `dnd-kit` 기반으로 탭과 작업 모두 드래그 앤 드롭 정렬을 지원합니다.
- 탭 삭제와 작업 삭제 후 `Sonner` 토스트를 통해 5초 이내 복구할 수 있습니다.
- 데스크톱과 모바일 환경에 맞춰 사이드바와 메인 영역이 반응형으로 동작합니다.
- 탭 단위 분류를 통해 성격이 다른 루틴과 반복 작업을 분리해서 관리할 수 있습니다.

<br />

**UI / 경험**

- Tailwind CSS와 shadcn/ui 기반 공통 UI를 사용해 일관된 인터페이스를 유지했습니다.
- hydrate 전 전체 렌더링을 막지 않도록 Skeleton UI를 적용해 필요한 영역만 점진적으로 로딩되도록 구성했습니다.
- 입력 과정에서 비활성화 처리와 상태 정규화를 적용해 잘못된 입력 흐름을 줄였습니다.

## 🧩 로컬 실행 방법

1. **저장소 클론**

   ```bash
   git clone https://github.com/carokann1945/carokann
   cd carokann
   ```

<br />

2. **환경변수 설정**
   `.env` 파일을 생성하고 아래 값을 채웁니다.

   ```bash
   NEXT_PUBLIC_SUPABASE_URL=your_supabase_url
   NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=your_supabase_publishable_key
   NEXT_PUBLIC_SITE_URL=http://localhost:3000
   DATABASE_URL=your_database_url
   DIRECT_URL=your_direct_database_url
   ```

<br />

3. **패키지 설치 및 개발 서버 실행**

   ```bash
   pnpm install
   pnpm dev
   ```

<br />

4. **추가 검증 명령**

   ```bash
   pnpm build
   pnpm lint
   ```

## ⚙️ 기술적 고민 및 트러블슈팅

1. **반복 작업의 진척도를 어떻게 눈에 보이게 유지할 것인가**

   기존 서비스들처럼 반복 작업을 완료하자마자 체크를 비워 버리면 사용자는 이번 사이클 안에서 얼마나 진행했는지를 확인하기 어렵습니다. Carokann은 체크 기록을 다음 리셋 시점까지 유지하고 목표 횟수와 완료 수를 함께 보여주는 방식으로 이 문제를 해결했습니다. 여기에 Temporal API를 도입해 월말, 윤년, 시간대 차이처럼 단순 `Date` 처리로 흔들리기 쉬운 경계 조건까지 안정적으로 계산하도록 설계했습니다.

<br />

2. **로컬 사용성과 로그인 저장을 하나의 흐름으로 연결**

   처음부터 로그인만 강제하면 진입 장벽이 높아지고 반대로 로컬 저장만 지원하면 데이터 저장이 보장되지 않습니다. 그래서 비회원 모드에서는 로컬 스토리지에 바로 저장하고 최초 로그인 시에는 기존 데이터를 Supabase DB로 이관하는 흐름을 만들었습니다. 초기 렌더링 과정에서 상태 동기화 로직이 분산되어 동일한 POST 요청이 중복 발생하던 구조도 정리해 요청 횟수를 `5회 → 2회`로 줄였고, `0.8초` 디바운스 저장과 `pagehide` 시점 flush, 클라이언트/서버 스냅샷 비교를 통해 불필요한 DB write까지 줄였습니다.

<br />

3. **hydrate 전 전체 렌더링 블로킹 없이 점진적으로 보여주기**

   초기 상태를 불러오는 동안 화면 전체를 막아 버리면 사용자는 앱이 멈춘 것처럼 느끼기 쉽습니다. 이를 줄이기 위해 탭 목록과 작업 목록에 Skeleton UI를 적용하고, hydrate가 끝난 영역부터 점진적으로 실제 UI를 보여주도록 구성했습니다. 덕분에 로딩 중에도 화면 구조를 예측할 수 있고 필요한 영역만 자연스럽게 전환되도록 UX를 개선할 수 있었습니다.

<br />

4. **복잡해지는 상태와 UI 로직을 어떻게 관리할 것인가**

   반복 작업, 로컬 저장, 서버 저장, 삭제 복구, 반응형 사이드바, 드래그 앤 드롭이 한 화면에 모이면 로직이 복잡하게 얽히기 시작합니다. Carokann은 feature 구조를 적용해 화면, 상태, 저장 책임을 분리했고, 중복되는 Dialog UI와 삭제 및 반응형 관련 로직 일부를 공통 컴포넌트와 커스텀 훅으로 나눠 유지보수성을 높였습니다. 여기에 입력값 검증, 범위 교정, 비활성화 처리, 상태 정규화를 더해 잘못된 데이터가 저장되지 않도록 방어했고 다양한 디바이스 환경에서도 같은 흐름을 유지하도록 반응형 레이아웃으로 설계했습니다.
